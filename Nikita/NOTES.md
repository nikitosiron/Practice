## Этап 1.1 — базовые понятия

### Node.js

Среда выполнения JavaScript **вне браузера**. Раньше JS жил только в браузере
(нажал кнопку → поменялось что-то на странице). Node.js взял движок V8 из
Chrome и сделал так, что JS может работать как обычная серверная программа:
читать файлы, слушать сетевой порт, отвечать на HTTP-запросы.

Команда `node server.js` запускает файл `server.js` как программу.
У меня установлен Node.js v22.23.1.

### npm (Node Package Manager)

Менеджер пакетов. Это две вещи сразу:

1. Онлайн-реестр чужих библиотек — [npmjs.com](https://www.npmjs.com).
2. Утилита командной строки, которая ставит/удаляет/обновляет библиотеки.

Например, `npm install express` скачает Express и положит его в локальную
папку `node_modules/`, а в `package.json` запишет, что наш проект от него
зависит. Тогда любой другой человек, склонировав репозиторий и сделав
`npm install`, получит ровно такой же набор библиотек.

У меня установлен npm 10.9.8.

### Express

Библиотека-фреймворк для написания веб-серверов на Node.js.

Чистый Node умеет принимать HTTP-запросы через встроенный модуль `http`,
но писать на нём напрямую муторно — много низкоуровневого кода (парсинг
тела запроса, маршрутизация руками). Express даёт удобный API:

```js
app.get('/api/data', (req, res) => {
  res.json({ ok: true });
});
```

— и маршрут готов.

**Альтернативы:** Fastify (быстрее), Koa (минималистичнее), Hapi (богаче
конфигурацией), или вообще без фреймворка на встроенном `http`.
**Почему Express:** самый распространённый, простой, много туториалов —
идеален для обучения.

### `package.json` — построчно

```json
{
  "name": "nikita",                 // имя проекта; взято из имени папки
  "version": "1.0.0",               // дефолт npm
  "description": "",                // без -y npm бы спросил

```

**Что значит `^5.2.1`** (caret-range, «домик»):
- Разрешено всё, что `>= 5.2.1` и `< 6.0.0`.
- То есть будущий `npm install` подтянет любую `5.x.y` — патчи и минорные
  обновления, но не «мажор» 6.x (где могут быть несовместимые изменения).
- Это [семвер (SemVer)](https://semver.org/lang/ru/): `MAJOR.MINOR.PATCH`.

**Что в `package.json` сгенерировалось автоматически:**
- `name` — из имени папки (`Nikita` → `nikita`, npm приводит к нижнему регистру).
- `version`, `main`, `license` — фиксированные дефолты npm.
- `description`, `author`, `keywords` — пустые, потому что флаг `-y`.
- `dependencies.express` — это **не** `npm init`, а `npm install express`.

### `package-lock.json`

Точные версии **всех** пакетов (включая транзитивные — то, что Express тянет
внутри себя). Гарантирует, что у меня, у Егора и у Ивана при `npm install`
установятся **ровно одинаковые** версии. Руками не редактируется.

### `node_modules/`

Папка с исходниками установленных библиотек. Тысячи файлов, десятки МБ.
**В git не коммитим** — на этапе 1.3 добавим её в `.gitignore`. Любой,
кто клонирует репозиторий и сделает `npm install`, получит её сам.

### Express 5 vs Express 4

У меня встал Express **5.2.1** (последняя стабильная). Большинство учебников
в интернете — про Express 4. Для нашего проекта это незаметно: `app.get`,
`app.use`, `express.static`, `express.json` работают одинаково. Если наткнусь
на несовместимость — отмечу здесь.

## Этап 1.3 — `scripts` и `.gitignore`

### Поправил `package.json`

```json
"main": "server.js",
"scripts": {
  "start": "node server.js",
  "test": "node --test"
}
```

**`scripts`** — словарь «короткое имя → команда». Запускается через
`npm run <имя>`. Зачем: чтобы команды запуска у всей команды были одинаковыми
(`npm start` работает у каждого одинаково, независимо от ОС и редактора).

- `start` и `test` — «магические» имена: можно писать `npm start` и
  `npm test` без `run`. Для своих скриптов всегда `npm run <имя>`.
- `node --test` — встроенный тест-раннер Node.js. Сам найдёт `*.test.js`
  и папку `tests/`. Будем использовать на Дне 3–4 (без jest/mocha).
- `main: "server.js"` — поле «главный файл проекта». Для приложения почти
  декоративное (важно, если кто-то импортирует проект как библиотеку).

**Альтернатива для `start`:** `nodemon` — авто-перезапуск сервера при
изменении файлов. Сейчас не ставим — полезнее самому видеть цикл
старт/стоп (Ctrl+C → `npm start`).

### Создал `.gitignore`

Текстовый файл в корне. Каждая строка — паттерн, который git **не** будет
отслеживать. Главное, что туда попало:

- `node_modules/` — иначе репозиторий распухнет на десятки МБ мусора.
- `*.log`, `npm-debug.log*` — логи Node.
- `.DS_Store`, `Thumbs.db` — системный мусор macOS / Windows.
- `.vscode/`, `.idea/` — личные настройки IDE.
- `.env` — на будущее, для секретов.

### Грабли: комментарии в `.gitignore`

Изначально я записал так:
```
.DS_Store          # macOS
Thumbs.db          # Windows
```

**Это неправильно.** В `.gitignore` `#` означает комментарий **только если
стоит в начале строки**. В строке с паттерном `#` — обычный символ, и git
будет искать файл с буквальным именем `.DS_Store          # macOS`.
То есть эти две строки ничего не игнорировали.

Правильно: комментарий **на отдельной строке** перед паттернами.

```
# .DS_Store — macOS, Thumbs.db — Windows
.DS_Store
Thumbs.db
```

## Этап 1.4 — `data/data.json`

### Зачем папка `data/`

Логически отделяем «данные» от «кода». Бэкенд-код будет в корне и в `src/`,
данные — в `data/`. Это конвенция, которой придерживается большинство
проектов с файловым хранилищем.

### Контракт (зафиксирован с командой)

```js
hero:        { title, stats: [{ value, label }] }
team[]:      { id, name, position, photo, vk, telegram }
vacancies[]: { id, title, format, url }
benefits[]:  { id, title, description }
```

- `id` — числа (1, 2, 3...). На Дне 3 напишем `getNextId`, который
  возьмёт максимум и прибавит 1.
- `photo` — путь относительно корня сайта (`travelline_site/`). Браузер
  загрузит `http://localhost:3000/upload/...`, потому что `express.static`
  будет раздавать `travelline_site/` как корень.
- `vk`, `telegram` — поставил `"#"` (заглушка), как в оригинальной странице
  ссылки тоже placeholder.

### JSON-синтаксис — основные правила

- Ключи объекта — всегда в **двойных** кавычках: `"name": "..."`.
- Строки — `"..."`, числа — без кавычек.
- **Никаких висячих запятых** после последнего элемента
  (`["a", "b",]` — невалидный JSON). Самая частая ошибка `SyntaxError`.
- Комментарии запрещены — JSON их не поддерживает.

### Откуда взял реальный контент

- `hero.title` и `hero.stats` — из `travelline_site/index.html`, секции
  `.hero` и `.advantages` (строки 147–219).
- `team` — три первых карточки из `.team__list` (строки 290–415).
  Фото — из `upload/resize_cache/...`, имена/должности — оттуда же.
- `vacancies` — три первых из `.vacancies__list` (строки 1828–1895),
  ссылки на hh.ru настоящие.
- `benefits` — три из `.bonus__list` (строки 2152–2186): «Поездки и
  обучение», «Здоровье и спорт», «Бонусы».

### Проверка валидности

Проверил, что JSON парсится:

```powershell
$d = Get-Content data\data.json -Raw | ConvertFrom-Json
$d.team.Count    # 3
```

Если ошибиться в синтаксисе (запятая, кавычка) — `ConvertFrom-Json` упадёт
с понятным сообщением. То же самое сделает `JSON.parse` в Node.

## Этап 1.5 — `server.js` (минимальная версия)

### Финальный код

```js
const express = require('express');
const path = require('path');
const fs = require('fs');

const app = express();
const PORT = 3000;
const DATA_PATH = path.join(__dirname, 'data', 'data.json');

app.use(express.json());
app.use(express.static(path.join(__dirname, 'travelline_site')));

app.get('/api/data', (req, res) => {
    try {
        const raw = fs.readFileSync(DATA_PATH, 'utf-8');
        const data = JSON.parse(raw);
        res.json(data);
    } catch (err) {
        console.error('Ошибка чтения data.json:', err);
        res.status(500).json({ success: false, message: 'Не удалось прочитать данные' });
    }
});

app.listen(PORT, () => {
    console.log(`Server running at http://localhost:${PORT}/`);
});
```

### Блок 1 — импорты модулей

```js
const express = require('express');
const path = require('path');
const fs = require('fs');
```

- **`require(имя)`** — функция Node, которая подгружает модуль и возвращает
  то, что модуль экспортирует. Это система модулей **CommonJS**.
- `express` ищется в `node_modules/`. Это та библиотека, которую мы
  поставили `npm install express`.
- `path` и `fs` — **встроенные** модули Node, ничего ставить не надо.
  `path` — для работы с путями файлов, `fs` (file system) — для чтения
  и записи файлов.
- **`const`** — переменная, которую нельзя переназначить. Привычка
  объявлять импорты через `const` помогает не наделать ошибок.

**Альтернатива:** ES Modules — `import express from 'express'`. Не используем,
потому что для CommonJS не нужна настройка в `package.json`.

### Блок 2 — создание приложения и константы

```js
const app = express();
const PORT = 3000;
const DATA_PATH = path.join(__dirname, 'data', 'data.json');
```

- `express()` — вызываем функцию `express` без аргументов; возвращает
  **объект-приложение**. Принято называть `app`.
- `PORT = 3000` — на каком TCP-порту слушать. 3000 — традиционный порт
  для локальной разработки Node-приложений.
- **`__dirname`** — встроенная переменная Node: абсолютный путь к папке,
  где лежит **текущий файл** (`server.js`). У меня это
  `D:\Work\Programming\Practice\Nikita`.
- **`path.join(...)`** — склеивает части пути правильным разделителем
  (`\` на Windows, `/` на Linux). Результат: `D:\...\Nikita\data\data.json`.

**Почему важно использовать `__dirname` + `path.join`, а не строку
`'data/data.json'`:** относительный путь резолвится от **текущей рабочей
директории процесса** (`process.cwd()`), а не от папки с файлом. Если
запустить `node Nikita/server.js` из родительской папки — путь сломается.
С `__dirname` файл всегда найдём, где бы ни запустили процесс.

### Блок 3 — middleware

```js
app.use(express.json());
app.use(express.static(path.join(__dirname, 'travelline_site')));
```

**Middleware** — функция, которая выполняется **между** входящим запросом
и финальным ответом. Может посмотреть/изменить `req` и `res` или передать
управление дальше по цепочке. `app.use(...)` цепляет middleware на все запросы.

- **`express.json()`** — middleware, которое смотрит на заголовок
  `Content-Type: application/json` и парсит тело запроса (`JSON.parse`),
  кладёт результат в `req.body`. На Дне 1 не пользуемся, но цепляем
  сразу — пригодится для PUT/POST на Дне 3.
- **`express.static(папка)`** — для каждого запроса смотрит, не лежит ли
  в указанной папке файл с подходящим именем. Если да — отдаёт его и
  **обрывает** цепочку. Если нет — пропускает дальше.

**Порядок middleware важен!** `express.json()` должен быть **до** маршрутов,
которые читают `req.body` — иначе `req.body` будет `undefined`. Express
обрабатывает middleware сверху вниз.

**Что отдаст `express.static('travelline_site')`:**
- `GET /` → `travelline_site/index.html` (по умолчанию отдаёт `index.html`)
- `GET /local/.../style.css` → `travelline_site/local/.../style.css`
- `GET /upload/.../photo.png` → `travelline_site/upload/.../photo.png`

### Блок 4 — маршрут `GET /api/data`

```js
app.get('/api/data', (req, res) => {
    try {
        const raw = fs.readFileSync(DATA_PATH, 'utf-8');
        const data = JSON.parse(raw);
        res.json(data);
    } catch (err) {
        console.error('Ошибка чтения data.json:', err);
        res.status(500).json({ success: false, message: 'Не удалось прочитать данные' });
    }
});
```

- **`app.get(путь, обработчик)`** — регистрирует маршрут. «Когда придёт
  HTTP-запрос **GET** на `/api/data` — вызови вторым аргументом функцию».
- **`(req, res) => { ... }`** — стрелочная функция, наш обработчик.
  - `req` (request) — объект с данными запроса: URL, метод, заголовки,
    тело (`req.body`), параметры пути (`req.params`), query-параметры
    (`req.query`).
  - `res` (response) — объект, через который **отвечаем** клиенту.

- **`fs.readFileSync(DATA_PATH, 'utf-8')`** — синхронно читает файл и
  возвращает **строку** (`'utf-8'` говорит «декодируй байты как UTF-8»).
  Без второго аргумента вернулся бы `Buffer` — сырые байты.

- **`JSON.parse(raw)`** — превращает строку JSON в JS-объект.
- **`res.json(data)`** — отправляет объект как JSON, сам выставляет
  заголовок `Content-Type: application/json`.

- **`try / catch`** — если в `try`-блоке что-то падает (нет файла, битый
  JSON), управление прыгает в `catch`. Без этого упало бы всё приложение.
  - `console.error(...)` — пишет в **stderr** (красным в терминале),
    чтобы было видно в логах.
  - `res.status(500).json(...)` — цепочка вызовов: `status(500)` ставит
    HTTP-код «500 Internal Server Error» и возвращает тот же `res`,
    к которому сразу вызываем `.json(...)`. Так задаём и код, и тело.
  - Формат тела — по согласованному с командой контракту ошибок:
    `{ success: false, message: "..." }`.

**Почему синхронное чтение, а не асинхронное:**
- На Дне 1 проще — никаких `async`/`await`.
- Файл маленький, читается мгновенно.
- На Дне 2 (`src/storage.js`) перейдём на `fs.promises.readFile` —
  объясню тогда, почему это правильнее для боевого сервера.

### Блок 5 — старт сервера

```js
app.listen(PORT, () => {
    console.log(`Server running at http://localhost:${PORT}/`);
});
```

- **`app.listen(порт, колбэк)`** — говорит Express «начни слушать порт
  3000». Сервер запускается и продолжает работать, пока процесс не убьют
  (Ctrl+C).
- Колбэк (второй аргумент) выполняется **один раз**, когда сервер успешно
  поднялся. Удобно для лог-сообщения «сервер на воздухе».
- **Бэктики `` ` ``** — шаблонная строка. Только в них работает `${PORT}`
  — подстановка значения переменной. С одинарными или двойными
  кавычками `${PORT}` напечатался бы буквально.

### Грабли этого этапа — 4 классических бага

1. **Лишняя скобка в стрелочной функции:**
   ```js
   (req, res)) => { ... }   // ❌ — лишняя )
   (req, res) => { ... }    // ✅
   ```
   Колбэк — это **одна** конструкция: `(параметры) => { тело }`. Скобки
   вокруг параметров и стрелка — части синтаксиса функции, не отдельные пары.

2. **Точка с запятой в объекте:**
   ```js
   { success: false; message: "..." }   // ❌
   { success: false, message: "..." }   // ✅
   ```
   В JS-объекте свойства разделяются **запятыми**. `;` — для операторов.

3. **Незакрытая скобка `app.get(`:**
   ```js
   app.get('/api/data', (req, res) => {
       ...
   };                                    // ❌ — закрыли только функцию
   });                                   // ✅ — `}` функция, `)` app.get, `;` оператор
   ```
   На каждое `(` должно быть `)`. У `app.get(` открывается **одна**
   скобка, и закрыть её надо в самом конце.

4. **Одинарные кавычки вместо бэктиков:**
   ```js
   '...${PORT}...'   // ❌ — выведет литерально ${PORT}
   `...${PORT}...`   // ✅ — подставит значение
   ```
   Подстановка `${...}` работает **только** в шаблонных строках
   (бэктики). Бэктик на клавиатуре — слева от единички, под Esc.

### Бонус: `node --check`

## Этап 2.2 — теория: асинхронность

### Node.js — один поток + event loop

Node выполняет мой JS-код в **одном потоке** (single-threaded). Не «поток на
каждый запрос», как классические Java/PHP-серверы. Тысячи подключений один
поток обслуживает за счёт **event loop** (цикл событий).

**Аналогия — официант на весь зал:**
- *Плохой (синхронный):* принял заказ → пошёл на кухню → **стоит и ждёт**,
  пока приготовят → принёс → пошёл к следующему. Пока ждёт — зал простаивает.
- *Хороший (асинхронный):* принял заказ → отдал на кухню → **сразу** к
  следующему столику. Кухня крикнет «готово!» — заберёт и отнесёт.

Event loop — это «хороший официант» внутри Node: асинхронную операцию
(чтение файла) он отдаёт операционной системе и **сразу** идёт обрабатывать
следующее событие. Когда ОС сообщает «готово», event loop в свободный момент
вызывает наш код-обработчик.

### I/O и почему оно «медленное»

**I/O** = Input/Output — любое общение с внешним миром: файлы, сеть, БД.
По меркам процессора это медленно (чтение файла с SSD ~0.1 мс, а за это
время CPU успел бы сотни тысяч операций). Если поток в это время просто
**стоит и ждёт** — время потеряно.

### Синхронный (`*Sync`) = блокирует поток

```js
const raw = fs.readFileSync(DATA_PATH, 'utf-8');  // ← весь сервер заморожен
const data = JSON.parse(raw);                      // выполнится только после
```

Пока файл не прочитан, **никакой другой запрос обработать нельзя**. На нашем
учебном проекте незаметно (файл крошечный, один пользователь). Но 100
одновременных запросов с чтением по 5 мс встанут в очередь → последний ждёт
~500 мс. Асинхронно — все 100 чтений уходят на ОС «параллельно», ~5–10 мс.

**Когда `*Sync` можно:** одноразовый код при старте (чтение конфига до
`app.listen`), CLI-скрипты. **В обработчике маршрута — никогда.**

### Три стиля асинхронности (исторически)

**1. Callbacks (старый стиль).** Передаём функцию «вызови, когда готово»:
```js
fs.readFile(p, 'utf-8', (err, raw) => { ... });   // err первым аргументом
```
Проблема — **callback hell**: несколько шагов подряд вкладываются пирамидой,
обработку ошибок пишешь на каждом уровне.

**2. Promise.** Объект-обёртка над «значением, которое будет позже».
Три состояния: **pending** (в процессе) → **fulfilled** (успех, есть
результат) / **rejected** (упало с ошибкой).
```js
fs.promises.readFile(p, 'utf-8')
    .then(raw => JSON.parse(raw))
    .catch(err => { ... });          // один catch на всю цепочку
```

**3. async/await (используем это).** Синтаксический сахар над Promise: код
выглядит как обычный синхронный.
```js
async function getData() {
    const raw = await fs.promises.readFile(p, 'utf-8');
    return JSON.parse(raw);
}
```

- **`async`** перед функцией: (1) разрешает `await` внутри; (2) функция
  **всегда возвращает Promise**, даже если внутри `return объект`.
- **`await`** перед Promise: приостанавливает **только эту функцию**, пока
  Promise не разрешится, и подставляет его результат. **Event loop при этом
  свободен** — `await` НЕ блокирует поток (в отличие от `*Sync`)!
- Если Promise упадёт (rejected) — `await` **бросит исключение**, которое
  ловится обычным `try/catch`. В этом вся красота: асинхронный код и ошибки
  обрабатываются так же, как синхронный.

### В Express 5

Express 5 умеет асинхронные обработчики (`async (req, res) => {}`) и сам
ловит исключения из них. В Express 4 нужны были обёртки — у нас этой боли нет.

## Этап 2.3 — `src/storage.js`

### Зачем отдельный модуль (разделение ответственности)

- `server.js` отвечает за **HTTP** (принять запрос, отдать ответ).
- `storage.js` отвечает за **хранение** (где данные и как их прочитать/записать).
- Завтра перейдём с JSON-файла на SQLite/Postgres → поменяется только
  `storage.js`, `server.js` не заметит.
- **Тестируемость:** на Дне 3 функцию `readData()` легко вызвать в тесте
  напрямую. Тестировать «маршрут, внутри которого читается файл» — гораздо
  сложнее.
- **Переиспользование:** на Дне 3 `dataService.js` будет звать и `readData`,
  и `writeData` для CRUD — без копипаста.

### Что такое модуль и `module.exports`

Каждый `.js`-файл в Node — **отдельный модуль**. Его переменные **не видны**
снаружи. Чтобы «вытащить» что-то наружу, кладём в специальный объект
`module.exports`; снаружи подключают через `require()`, который возвращает
содержимое `module.exports`.

```js
// math.js
module.exports = { add, mul };

// app.js
const { add, mul } = require('./math');   // деструктуризация
```

- `require('./math')` — точка = «из текущей папки». Без точки
  (`require('express')`) — ищется в `node_modules/`.
- Расширение `.js` можно опустить.
- `require` **кеширует** модуль: повторный `require` того же файла не
  выполняет его заново, отдаёт тот же объект.
- **Ловушка:** `exports = {...}` (без `module.`) НЕ работает — переназначает
  локальную переменную, наружу ничего не уедет. Только `module.exports = {...}`.
- Запись `{ readData, writeData }` — **shorthand** для
  `{ readData: readData, writeData: writeData }` (имя свойства = имя переменной).

### Финальный код

```js
const fs = require('fs/promises');
const path = require('path');

const DATA_PATH = path.join(__dirname, '..', 'data', 'data.json');

async function readData() {
    const raw = await fs.readFile(DATA_PATH, 'utf-8');
    return JSON.parse(raw);
}

async function writeData(data) {
    const json = JSON.stringify(data, null, 2);
    await fs.writeFile(DATA_PATH, json, 'utf-8');
}

module.exports = { readData, writeData };
```

### Разбор по блокам

- **`require('fs/promises')`** — у модуля `fs` три API: синхронный
  (`readFileSync`), колбэк-стиль (`readFile(..., cb)`) и Promise-стиль.
  `fs/promises` даёт **только** Promise-версии (те же имена, но возвращают
  Promise, работают с `await`). Короче, чем `require('fs').promises`, и явно
  заявляет «тут только асинхронный API».
- **`path.join(__dirname, '..', 'data', 'data.json')`** — `__dirname` теперь
  папка `src/`; `'..'` поднимает на уровень вверх (в `Nikita/`), дальше
  `data/data.json`. Каждый сегмент отдельным аргументом → портабельно
  (Windows `\`, Linux `/`).
- **`readData`** — `fs.readFile` сразу возвращает pending-Promise; `await`
  ждёт его, подставляет **строку** содержимого; `JSON.parse` → объект;
  `return` объекта из `async`-функции наружу прилетит как `Promise<объект>`
  (получить через `await readData()`).
- **`writeData`** — `JSON.stringify(data, null, 2)`: `null` = replacer (фильтр
  свойств, не нужен), `2` = отступ в 2 пробела для читаемости и **построчных
  git-диффов**. `fs.writeFile` **перезаписывает файл целиком**, возвращает
  `Promise<undefined>` (нужна ради побочного эффекта).
- **Почему без `try/catch` внутри:** ошибку пропускаем **наверх** — её
  обработает маршрут, где есть `res` для ответа `500`. Принцип: *низкий слой
  пропускает ошибку, верхний решает, что делать*. В `async`-функции это
  происходит само: `await` бросит → Promise станет rejected → наверху поймает
  `try/catch`.

### Грабли 2.3: `requires` вместо `require`

Написал `const path = requires('path');` (лишняя `s`).
- `require` — встроенная функция Node, **без `s`**. `requires` не существует.
- **`node --check` это НЕ ловит** — синтаксически `requires('path')` валиден
  (просто «вызов функции с именем requires»). Ошибка вылезет только при
  **выполнении**: `ReferenceError: requires is not defined`.
- Падает при старте сервера (модуль подключается один раз) — это даже хорошо,
  лучше сразу, чем при первом запросе.

## Этап 2.4 — подключение в `server.js`

### Что изменилось

```js
const express = require('express');
const path = require('path');
const { readData } = require('./src/storage');   // ← новый импорт

const app = express();
const PORT = 3000;

app.use(express.json());
app.use(express.static(path.join(__dirname, 'travelline_site')));

app.get('/api/data', async (req, res) => {        // ← async

});

});
```

- `const fs = require('fs');` — `fs` использовался только ради

- `const DATA_PATH = ...` — путь к файлу теперь знает **только** `storage.js`.
  HTTP-слой не должен знать, *где* физически лежат данные. Оставить здесь —
  «протечка» детали хранения в чужой слой.

**Что оставили:**
- `const path = require('path');` — всё ещё нужен для
  `express.static(path.join(...))` (раздача статики — законная HTTP-обязанность).

**Что в маршруте:** `async` перед `(req, res)`; две строки чтения схлопнулись
в `const data = await readData();`. `try/catch` и формат `500` — без
изменений (`await` бросает при ошибке → ловит тот же `catch`).

**Импорт `{ readData }`, а не `writeData`:** `writeData` в `server.js` пока
негде звать (мутации — на Дне 3). Импортировать неиспользуемое — лишнее.

| Модули | `module.exports = {...}`, деструктуризация в `require`, кеширование `require`, shorthand `{ a, b }`, ловушка `exports =` без `module.` |
| Архитектура | разделение ответственности (слой хранения ≠ HTTP-слой), «низкий слой пропускает ошибку наверх» |
| Express | `async (req, res) => {}` + `await` в обработчике, Express 5 сам ловит async-ошибки |

### Грабли, которые поймал

1. `requires('path')` вместо `require` — `node --check` НЕ ловит (валидный
   синтаксис), падает при выполнении `ReferenceError`.
2. (запомнить на Дню 3) `fs.writeFile` перезаписывает файл **целиком**; два
   одновременных `writeData` → race condition. Для учебного проекта (один
   админ) не страшно; в проде — блокировки или БД.

## Этап 3.1 — REST, `req.body`, `req.params`, коды, валидация

### HTTP-методы (REST)

URL = ресурс, метод = что с ним сделать. `/api/team` — коллекция,
`/api/team/2` — конкретный сотрудник.

| Метод | Намерение | Тело? |
|---|---|---|
| GET | прочитать, не меняя | нет |
| POST | создать новый | да |
| PUT | заменить/обновить существующий | да |
| DELETE | удалить | обычно нет |

GET не должен менять данные; POST/PUT/DELETE — меняют. В Express:
`app.get/post/put/delete(путь, обработчик)`.

### `req.body` — тело запроса

Распарсенное тело POST/PUT. Работает **только** при подключённом
`app.use(express.json())` (повесили ещё в Дне 1) — он парсит JSON-тело в
объект и кладёт в `req.body`. Без него `req.body === undefined`.
**`req.body` приходит от клиента — доверять нельзя** → валидация.

### `req.params` — переменные пути (`:id`)

```js
app.put('/api/team/:id', (req, res) => {
    const id = req.params.id;   // "2" — ВСЕГДА СТРОКА!
});
```
`:id` в пути объявляет переменную; `req.params` = `{ id: "2" }`.
**Грабли:** `req.params.id` — строка, а `id` в данных — число. `2 === "2"`
→ `false`. Нужно `Number(req.params.id)`.
(`req.query` — это `?key=value`, нам пока не нужен.)

### Коды ответов

| Код | Значение | Когда |
|---|---|---|
| 200 OK | успех | GET, удачные PUT/DELETE |
| 201 Created | создан ресурс | удачный POST |
| 400 Bad Request | плохие данные клиента | провал валидации |
| 404 Not Found | ресурс не найден | нет такого id |
| 500 Internal Server Error | сервер сам сломался | ошибка файла |

**4xx — виноват клиент, 5xx — виноват сервер.** По коду фронт решает, что
показать пользователю.

### Валидация

Проверка `req.body` до записи: поля на месте, нужного типа, не пустые.
Плохо → `400`, не пишем в файл. **Где:** в `dataService` (правила «что такое
валидный сотрудник» — про предметную область, не про HTTP).

## Этап 3.2 — `getNextId` и `updateHero`

### Зачем третий слой (`dataService`)

```
HTTP (server.js) → бизнес-логика (dataService.js) → хранение (storage.js) → data.json
```
Каждый слой знает только соседа снизу. `storage` — «тупой» (читай/пиши весь
файл), `dataService` — правила (валидация, id, CRUD), `server` — HTTP.
Классика: **controller → service → repository**. Без слоя получился бы
«толстый контроллер» с дублированием и нетестируемой логикой.

### `getNextId(list)` — чистая функция

```js
function getNextId(list) {
    if (list.length === 0) return 1;
    return Math.max(...list.map(item => item.id)) + 1;
}
```
- **`.map(item => item.id)`** — метод массива: из `[{id:1},{id:2}]` делает
  новый массив `[1, 2]`.
- **`Math.max(a, b, c)`** — макс из аргументов (через запятую, не массив!).
- **`...` (spread)** — раскрывает массив в аргументы: `Math.max(...[1,2,3])`
  = `Math.max(1,2,3)`. Без спреда `Math.max([1,2,3])` = `NaN`.
- **Пустой массив отдельно** → `1` (иначе `Math.max()` = `-Infinity`).
- id **не переиспользуются**: из `[1,3]` следующий — `4`, не `2`.
- **Чистая функция** (нет I/O, нет побочных эффектов) — легче всего тестировать.

### Сигнал об ошибке валидации: бросаем Error со `.status`

Выбрали **бросать исключение** (а не возвращать `{ok:false}`) — ложится на
существующий `try/catch` в маршрутах, продолжает принцип «низкий слой бросает,
верхний решает». К объекту `Error` дописываем своё поле `.status`:
```js
function validationError(message) {
    const err = new Error(message);
    err.status = 400;            // объекту Error можно дописать своё поле
    return err;
}
// бросаем на месте: throw validationError('...')
```
Маршрут потом разрулит по `err.status` (есть → 400/404, нет → 500).

### `updateHero(hero)` — паттерн мутации

```js
async function updateHero(hero) {
    if (!hero || typeof hero !== 'object') throw validationError('...');
    if (typeof hero.title !== 'string' || hero.title.trim() === '') throw validationError('...');
    if (!Array.isArray(hero.stats)) throw validationError('...');

    const data = await readData();                       // 1. прочитать ВСЁ
    data.hero = { title: hero.title, stats: hero.stats }; // 2. изменить кусок
    await writeData(data);                               // 3. записать ВСЁ
    return data.hero;
}
```
Паттерн файлового хранилища: **читать всё → менять в памяти → писать всё**
(`writeData` перезаписывает файл целиком — грабли Дня 2).

Валидация по блокам:
- **`!hero || typeof hero !== 'object'`** — `typeof null === 'object'`
  (причуда JS!), поэтому `!hero` отдельно отсекает `null`/`undefined`.
- **`.trim() === ''`** — срезает пробелы; ловит `""` и `"   "`. Делать
  **после** проверки на строку (иначе `.trim()` упадёт).
- **`Array.isArray(x)`** — единственный надёжный способ (`typeof []` =
  `'object'`).
- **`data.hero = { title:..., stats:... }`** — пересобираем из **проверенных**
  полей, а не `= hero`: лишние поля клиента (`hero.hacked`) не попадут в файл.

## Этап 3.3 — CRUD team

### Новые конструкции

- **`.find(pred)`** → сам объект или `undefined`; **`.findIndex(pred)`** →
  индекс или `-1`. Для update/delete нужен индекс → `findIndex`.
  `-1` = «не найдено» (соглашение JS).
- **`.push(x)`** — добавить в конец (для add).
- **`.splice(i, n)`** — вырезать `n` элементов с позиции `i`, **вернуть
  массив вырезанных**. `const [removed] = arr.splice(i, 1)` — деструктуризация.
- **Spread в объекте** `{ ...старый, поле: новое }` — копирует все поля,
  поверх кладёт новые. `id` сохраняется из `...старый`.
- **`??` (nullish coalescing)** — «левое, а если `null`/`undefined` — правое».
  Отличие от `||`: `??` НЕ срабатывает на `""`/`0`/`false`. Поэтому
  `photo: member.photo ?? старое` — пустую строку уважаем как осознанный выбор.

### Три функции

```js
function notFoundError(message) {   // как validationError, но status = 404
    const err = new Error(message);
    err.status = 404;
    return err;
}

function validateTeamMember(member) {   // зовут и add, и update — не дублируем
    if (!member || typeof member !== 'object') throw validationError('...');
    if (typeof member.name !== 'string' || member.name.trim() === '') throw validationError('...');
    if (typeof member.position !== 'string' || member.position.trim() === '') throw validationError('...');
}

async function addTeamMember(member) {
    validateTeamMember(member);
    const data = await readData();
    const newMember = {
        id: getNextId(data.team),     // id назначает СЕРВЕР, не клиент
        name: member.name,
        position: member.position,
        photo: member.photo ?? '',
        vk: member.vk ?? '#',
    };
    data.team.push(newMember);
    await writeData(data);
    return newMember;
}

async function updateTeamMember(id, member) {
    validateTeamMember(member);
    const data = await readData();
    const index = data.team.findIndex(m => m.id === id);
    if (index === -1) throw notFoundError(`Сотрудник с id=${id} не найден`);
    const updated = { ...data.team[index], name: member.name, position: member.position,
                      photo: member.photo ?? data.team[index].photo,
                      vk: member.vk ?? data.team[index].vk };
    data.team[index] = updated;
    await writeData(data);
    return updated;
}

async function deleteTeamMember(id) {
    const data = await readData();
    const index = data.team.findIndex(m => m.id === id);
    if (index === -1) throw notFoundError(`Сотрудник с id=${id} не найден`);
    const [removed] = data.team.splice(index, 1);
    await writeData(data);            // ← БЕЗ ЭТОГО удаление не сохранится!
    return removed;
}
```

### Грабли 3.3 (поймал при проверке моего кода)

1. **`deleteTeamMember` без `writeData`** — `splice` вырезал только из памяти,
   файл не перезаписан → после рестарта «удалённый» возвращается. Удаление
   фактически не работало. (Критический баг.)
2. **Пропала валидация `name`** — при правках по `telegram` случайно удалил
   блок проверки `name`; можно было создать сотрудника без имени.
3. **`id` приходит в сервис уже числом** — преобразование `"2"→2` и проверку
   `NaN` делает маршрут; сервис остаётся «чистым» от HTTP → легко тестировать.
4. **update — компромисс:** `name`/`position` обязательны, `photo`/`vk`
   опциональны (через `??` сохраняем старое). Строго REST PUT = полная замена,
   но для админки «досыл изменений» дружелюбнее.

## Этап 3.4 — маршруты в `server.js`

### Шаблон маршрута-мутации

```js
app.put('/api/hero', async (req, res) => {
    try {
        const updated = await updateHero(req.body);
        res.json({ success: true, data: updated });
    } catch (err) {
        console.error('PUT /api/hero:', err);
        const status = err.status || 500;
        const message = err.status ? err.message : 'Внутренняя ошибка сервера';
        res.status(status).json({ success: false, message });
    }
});
```

Сердце обработки ошибок (в каждом `catch`):
- **`err.status || 500`** — наши ошибки несут `status` (400/404); у прочих
  (файл, баг) поля нет → `500`. Так одно `catch` различает «вина клиента/сервера».
- **`err.status ? err.message : 'Внутренняя ошибка сервера'`** — для серверных
  ошибок НЕ показываем `err.message` (утечка внутренних деталей).
- **`err.message`** — текст, переданный в `new Error(текст)`.

### `:id`-маршруты

```js
const id = Number(req.params.id);
if (Number.isNaN(id)) {
    return res.status(400).json({ success: false, message: 'id должен быть числом' });
}
```
- **`Number(req.params.id)`** — строка → число (`"2"`→`2`).
- **`Number.isNaN(id)`** — `/api/team/abc` → `Number('abc')` = `NaN` → `400`.
  (`id === NaN` всегда `false`, так нельзя — только `Number.isNaN`.)
- **`return`** перед `res.status(400)` обязателен! Иначе код пойдёт дальше,
  ответит второй раз → «Cannot set headers after they are sent».

### 4 маршрута
`PUT /api/hero` (200), `POST /api/team` (**201** — создание),
`PUT /api/team/:id` (200), `DELETE /api/team/:id` (200). Порядок в файле:
статика → GET → мутации → `app.listen` (всегда последним).

### Долг (отметить на День 5)
`catch` дублируется в 5 маршрутах. Можно убрать через **error-handling
middleware** Express (`(err, req, res, next)`) — отрефакторим, когда будем
причёсывать единый формат ошибок для команды.

## Этап 3.5 — автотесты (`node --test`)

### Юнит-тест и `node:test`

**Юнит-тест** — программа, вызывающая один кусок кода с известным входом и
проверяющая выход. Ловит регрессии сами, документирует поведение, даёт
уверенность при рефакторинге. Используем **встроенный** раннер (без jest/mocha):
```js
const test = require('node:test');     // test(имя, fn)
const assert = require('node:assert'); // проверки
```
- **`test('имя', () => {...})`** — описывает тест; если внутри ничего не
  «бросило» — пройден (✓), иначе провален (✗).
- **`assert.strictEqual(actual, expected)`** — проверка `===`.
  (`deepStrictEqual` — для объектов/массивов «по содержимому».)
- **async-тест + `assert.rejects(fn, проверка)`** — ждёт, что Promise из
  `fn()` упадёт; вторым аргументом проверяем ошибку:
  ```js
  await assert.rejects(() => addTeamMember({ position: 'dev' }),
                       (err) => err.status === 400);
  ```

### Тонкость: тесты не должны портить `data.json`

`addTeamMember` с **валидными** данными перезапишет боевой файл. Поэтому на
Дне 3 тестируем только то, что файла **не трогает**:
1. **`getNextId`** — чистая функция (4 кейса).
2. **Пути валидации** — при плохом вводе функция бросает **до** `readData()`
   (порядок: сначала `validateTeamMember`, потом чтение). Безопасно (4 кейса).

Полные тесты с записью требуют **изоляции хранилища** (временный файл вместо
боевого) — отдельная техника, отложили на День 4/5.

### Где лежат / как запускать

`node --test` (наш `npm test`) сам находит файлы в папке **`test/`** и/или с
именами `*.test.js`. Положили в `test/dataService.test.js` (оба условия).
`require('../src/dataService')` — `..` т.к. тест в `test/`, сервис в `src/`.

### Результат

```
# tests 8   # pass 8   # fail 0
```
Формат вывода — **TAP** (Test Anything Protocol): `ok N - имя` = пройден,
`not ok N` = провален. Код возврата `npm test`: `0` при зелёных, не-`0` при
красных (пригодится для CI на GitHub). Проверили: прогон тестов `data.json`
не изменил (`git diff` — только удаление `telegram`).

## Этап 4.1 — копировать или обобщать (path A vs B)

`vacancies`/`benefits` CRUD почти идентичны `team`: отличаются только
коллекцией (`data.vacancies` vs `data.team`), набором полей и дефолтами.
Два пути:
- **A) Копировать (explicit)** — отдельные функции на каждую сущность.
  Плюс: читается «в лоб». Минус: дублирование.
- **B) Обобщить (factory)** — одна функция-фабрика принимает имя коллекции +
  правила и возвращает `add/update/delete`. Плюс: нет дублирования. Минус:
  абстрактнее, новичку труднее, баги ищутся сложнее.

**Выбрали A.** Причины: осваиваем базу (явный код важнее «красивого»);
сущностей всего три; обобщить можно потом, с зелёными тестами на подстраховке.

**Правило:** *«дублирование дешевле неправильной абстракции»* — пока не видишь
все случаи целиком, обобщать рано. Сначала заставь работать → потом сделай

### Проверка url через `new URL`

```js
try {
    new URL(vacancy.url);
} catch {
    throw validationError('Поле url должно быть корректной ссылкой ...');
}
```
- **`new URL(строка)`** — встроенный конструктор (Node + браузеры). Разбирает
  строку как URL; на валидной — создаёт объект (нам не нужен, важен сам факт),
  на кривой — **бросает `TypeError`**.
- **try/catch** превращает чужой `TypeError` в нашу `validationError` (400)
  с понятным текстом.
- **`catch { ... }` без `(err)`** — optional catch binding: когда объект ошибки
  не нужен, скобки можно опустить (Node 22 поддерживает).
- **Тонкость — требуется ПРОТОКОЛ:**
  - `new URL('https://hh.ru/vac/1')` → ок;
  - `new URL('hh.ru/vac/1')` → **бросит** (нет `https://`);
  - `new URL('просто текст')` → бросит.
  Значит админ обязан вставлять полные ссылки с `https://`. (Грабли: «почему
  не сохраняется вакансия» — забыл протокол.)

# День 5 — слияние в единый backend (интеграция с Ваней)

## Зачем

После `git fetch`/fast-forward подтянулась Ванина работа (коммит
`933105d firstworkingversion`): он в своей папке `Ivan/` добавил новую сущность
**`positions` (справочник должностей)** и флаг **`active`**. Получилось два
разных бэкенда: мой (`Nikita/`) и Ванин. Решили **свести всё в один** — мой
`Nikita/` становится единым backend'ом.

## Как определили, что переносить (diff вместо переписывания)

Сравнил файлы напрямую:
```bash
diff Nikita/src/dataService.js Ivan/src/dataService.js
diff Nikita/server.js          Ivan/server.js
```
Вывод: **Ванин backend — строгое надмножество моего**. Все различия — только
*добавления* (`>`), ничего из моего не удалено и не изменено. В частности, моя
`new URL`-проверка в `validateVacancy` у Вани присутствует один в один (его
бэкенд вырос из моего коммита Дня 4). Раз надмножество — проще всего взять его
файлы как основу, а не переносить куски руками.

**Правило:** перед слиянием — `diff`. Если одна сторона — надмножество другой,
слияние сводится к копированию + возврату локальных отличий. Переписывать
руками = плюс шанс на опечатку.

## Что нового принесла Ванина часть

### 1. Сущность `positions` (справочник должностей)
`{ id, title }`. Полный CRUD в `dataService.js`:
`validatePosition`, `getPositions`, `addPosition`, `updatePosition`,
`deletePosition`. Маршруты `GET/POST/PUT/DELETE /api/positions`.

🔑 **Проверка целостности (referential integrity)** в `deletePosition`:
```js
const usedBy = data.team.filter(m => m.position === data.positions[index].title);
if (usedBy.length > 0) {
    throw validationError('Нельзя удалить должность, которая используется ...');
}
```
Нельзя удалить должность, если она стоит хоть у одного сотрудника — иначе
данные «осиротеют». Это шаг за пределы простого CRUD: связь между сущностями.

### 2. Флаг `active` (мягкое вкл/выкл)
В `team`/`vacancies`/`benefits` добавлено поле `active`:
```js
active: member.active !== false   // нет поля -> считается активным (true)
```
Идея: не удалять запись, а «спрятать» (не показывать на публичной странице).
Логика `!== false` означает «активно по умолчанию»: старые записи без поля
считаются активными, выключить можно только явным `active: false`.

### 3. Продвинутая админка (`admin.js` 576 → 1067 строк)
Ванина версия — надмножество моей: секция «Должности», должность сотрудника
через **выпадающий список** (`<select>` из справочника, а не свободный текст),
инлайн «+ Новая должность», **табличный вид** сотрудников, **поиск**,
`active`-бейджи и тумблеры, единый разбор ответа `parseApiResponse`.

### Грабли 6.5 — старый `node` на порту 3000

**Урок.** Перед smoke-тестом всегда:
```powershell
Get-NetTCPConnection -LocalPort 3000 -State Listen
```
Если что-то висит — убить:
```powershell
$p = (Get-NetTCPConnection -LocalPort 3000 -State Listen).OwningProcess
Stop-Process -Id $p -Force
```
Через переменную. Прямой pipe в `Stop-Process` в PowerShell 5.1 падает на
`InputObjectNotBound` — там особенности биндинга параметров.

## Этап N.1 — контракт и сид

```js
timeline[]: {
  id,           // number
  year,         // number, 1990 ≤ year ≤ 2100, целое
  name,         // string, обязательное
  category,     // string, enum: "B2B" | "B2C" | "B2E"
  subtitle,     // string, обязательное
  description,  // string, обязательное

```

### Ключевые решения — обоснованы
- **Название `timeline`**, не `platform`/`products` — семантика «упорядочено
  по годам» понятна команде. `platform` — техническое имя секции в HTML.
- **`year` — число, не строка**: сортировка на клиенте через `a.year - b.year`
  без парсинга.
- **`year` диапазон 1990–2100** — отсекает опечатки типа `20` (пропущенная
  цифра) или `20200`.
- **`category` — enum**, не свободная строка. В HTML встречались смеси
  кириллицы и латиницы (`B2С`, `В2В` — легко не заметить глазом). Enum
  строгий + нормализация в сиде решает грязь на входе.
- **`subtitle`/`description` — оба обязательные.** Продукт без описания в БД
  бессмыслен. Единообразно с benefits.
- **Иконки НЕ в контракте.** Три альтернативы обсуждали: (A) SVG-строкой в
  JSON — раздувает файл; (B) `iconKey: "booking-engine"` + фронт-словарь;
  (C) фронт сам решает без поля. **Взял C.** Причина: бэк не должен диктовать
  визуал. Если Ваня решит нужно поле — миграция дешёвая (добавим `iconKey`
  строкой с дефолтом `""`).

### Сид — 3 записи
Взял три полные карточки (name + category + subtitle + description), чтобы
схема была видна на всех полях:
- id=1: **Booking Engine** (2009, B2C) — флагман, модуль бронирования.
- id=2: **PMS Integration** (2011, B2B) — интеграция с АСУ.
- id=3: **Analytics** (2012, B2B) — модуль аналитики.

Категории нормализовал к латинице, даже там где в HTML была кириллица.

## Этап N.2 — `dataService.js`

**Константа-enum** на уровне модуля:
```js
const ALLOWED_TIMELINE_CATEGORIES = ['B2B', 'B2C', 'B2E'];
```
Массив создаётся один раз при `require`, не при каждом POST. Наружу не
экспортирую (пока никто не просил).

**Порядок валидации в `validateTimelineItem`**:
1. Object.
2. `name` — обязательная строка.
3. `year` — одно `if` с четырьмя `||`:
   ```js
   if (typeof item.year !== 'number' || !Number.isInteger(item.year)
       || item.year < 1990 || item.year > 2100) throw ...
   ```
   `Number.isInteger` за один вызов ловит `2020.5`, `NaN`, `Infinity` —
   приведения типов не делает (`isInteger('2020')` → `false`). Порядок
   литералов важен: short-circuit защищает `isInteger` от вызова на
   `undefined`.
4. `category` — сначала проверка на строку (осмысленное сообщение об ошибке),
   потом `.includes` для enum. `.includes` использует `===` — регистр и
   алфавит важны: `"B2b"` не пройдёт, `"B2С"` (кириллица) не пройдёт. Это и
   нужно от enum.
5. `subtitle`, `description` — обязательные строки.

`add`/`update`/`delete` — три калька с benefit/gallery. Внутри —
defensive-init `if (!data.timeline) data.timeline = []` (как в positions,
не как в gallery — единообразие с более консервативным вариантом).
